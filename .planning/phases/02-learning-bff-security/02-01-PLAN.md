# 02-01 Plan: Mutation Proxy Sanitization

## Goal

Make learning JSON mutations SaaS-safe by construction so normal users cannot send provider/model/API-key/base URL fields through hosted ZAKI routes.

## Requirements

- LEARN-SEC-001
- LEARN-SEC-002
- LEARN-SEC-005
- LEARN-SET-001 through LEARN-SET-004

## Code Truth

- `backend/src/index.js`
- Existing sanitizer: `sanitizeLearningClientPayload`
- Existing risk: some JSON mutation proxies still forward `req.body` directly to the learning engine.

## Implementation Steps

1. Inventory every learning route that accepts JSON mutation bodies.
2. Classify routes as:
   - read-only/no body
   - safe direct body
   - sanitize recursively
   - explicit route allowlist required
3. Replace direct `req.body` forwarding on mutation paths with `sanitizeLearningClientPayload` or route-specific allowlists.
4. Add regression tests or route-level assertions for provider/model/API-key/base URL stripping.
5. Verify existing learning flows still work through the BFF.
6. Update `02-01-SUMMARY.md`.

## Acceptance Criteria

- No normal-user learning JSON mutation route forwards operator-managed fields unchanged.
- Recursive nested provider/model fields are stripped or rejected.
- Tests cover representative mutation paths.
- No provider/model/API-key user settings are introduced.
