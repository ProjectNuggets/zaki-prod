# 02-03 Plan: WebSocket Schema And Quota Hardening

Make learning WebSocket ingress schema-safe and quota-correct so passive sockets do not consume prompt quota and client messages cannot smuggle operator-managed routing fields.

## Implementation Steps

1. Review every learning WebSocket alias:
   - `/api/learning/ws`
   - `/api/learning/book/ws`
   - `/api/learning/chat/ws`
   - `/api/learning/solve/ws`
   - `/api/learning/vision/solve/ws`

   - `/api/learning/questions/generate/ws`
2. Replace root-level stripping assumptions with explicit root field allowlists for JSON messages.
3. Preserve recursive stripping of provider/model/API-key/base URL fields.
4. Verify quota is consumed only on prompt/mutating messages, not passive subscribe/progress sockets.
5. Add contract tests for:
   - allowed root payload
   - unknown root field removal
   - nested operator field removal
   - passive message quota exemption
   - prompt/mutating message quota consumption
6. Update `02-03-SUMMARY.md`.

## Acceptance Criteria

- WebSocket JSON messages are allowlisted at root and recursively sanitized.
- Passive progress/subscribe sockets do not consume prompt quota.
- Prompt/mutating messages still consume quota.
- Binary frames preserve existing behavior unless a route-specific policy changes.
