# 04-04 Backup/Restore Drill And Disaster Recovery Runbook

Date: 2026-05-07

## Goal

Make backup, restore, and disaster recovery status explicit before paid-user rollout.

## Code Truth

- Learning tenant data is stored under the downstream learning engine tenant root.
- ZAKI backend owns auth, quota, governance audit, account export/delete, retention policy, and operator gate status.
- A real restore drill requires production/staging backup target credentials and a snapshot location; code must expose whether that evidence exists.

## Implementation Steps

1. Add a disaster recovery policy resolver from operator environment variables.
2. Add explicit readiness gates for:
   - learning configuration
   - tenant data root
   - backups enabled
   - backup target
   - immutable learning engine image tag
   - fresh restore drill evidence
3. Expose super-admin DR status via `/api/internal/learning/disaster-recovery`.
4. Add a runbook for backup, restore drill, and disaster recovery.
5. Update the integration spec and planning state.
6. Verify with focused unit tests, lint, and full backend tests.

## Acceptance

- Production readiness cannot be claimed unless the DR endpoint reports ready.
- Missing backup target or stale restore drill evidence is visible as a failed gate.
- The runbook names the exact tenant root, database tables, restore verification checks, and release gate.
