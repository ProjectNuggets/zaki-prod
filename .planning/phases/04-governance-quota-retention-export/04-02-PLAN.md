# 04-02 Data Deletion, Export, And Audit State

Date: 2026-05-07

## Goal

Make learning account export and deletion governable for hosted SaaS by recording durable audit state for every user-triggered learning export/delete path.

## Existing Code Truth

- `/api/account/export` already includes a learning snapshot when the learning engine is configured.
- `/api/account/delete` already enumerates learning resources and fails closed if learning deletion cannot be enumerated or completed.
- The missing production gate is durable audit state: there was no persistent record of learning export/delete attempts, outcomes, request ids, or summaries.

## Implementation Steps

1. Add a non-PII learning account audit table keyed by user id plus hashed subject.
2. Add helpers to record and list learning account audit events.
3. Summarize learning export/delete outcomes without copying exported learning payloads into the audit ledger.
4. Record export success, completed-with-errors, and failure outcomes.
5. Record deletion `started` before destructive cleanup, then success or failure after the attempt.
6. Add `/api/account/learning/audit` for the signed-in user to inspect their audit state.
7. Cover the helper contract with unit tests and run the backend suite.

## Acceptance

- Export and deletion attempts leave durable audit events.
- Deletion records a `started` event before irreversible account cleanup.
- Audit details contain counts and resource types, not full learning content.
- Failed deletion attempts remain visible through audit state while the account still exists.
