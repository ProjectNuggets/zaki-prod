# 04-03 Retention And Cleanup Policies

Date: 2026-05-07

## Goal

Define and enforce the first hosted retention policy for ZAKI Learn without deleting active users' learning content unexpectedly.

## Decisions

- Active-account learning content is retained for the account lifetime.
- Account deletion remains the primary cleanup mechanism for uploaded sources, indexes, generated books, notebooks, sessions, tutor state, and saved artifacts.
- Audit rows are retained separately from account rows using non-PII subject hashes.
- Exact quota caps remain provisional until personal-user unit economics are validated.
- Retention is separate from quota: hitting a quota cap does not delete a user's learning data.

## Retention Matrix

| Data class | Retention | Enforcement |
| --- | --- | --- |
| Uploaded source files and indexes | Account lifetime | Account deletion path |
| Generated books, notebooks, sessions, tutor state, and memory | Account lifetime | Account deletion path |
| Generated artifacts referenced by saved records | Account lifetime | Account deletion path |
| Transient generated artifacts not attached to saved records | 90 days default | Operator cleanup policy |
| Stale task/progress state | 30 days default | Operator cleanup policy |
| Learning export/delete audit events | 730 days default | Scheduled ZAKI backend cleanup |

## Implementation Steps

1. Add a retention policy resolver with operator environment overrides.
2. Add a scheduled cleanup helper for expired learning audit events.
3. Expose super-admin retention status and manual cleanup endpoints.
4. Add tests for default policy, operator overrides, disabled cleanup, and bound SQL parameters.
5. Update the integration spec and GSD state.

## Acceptance

- The backend has explicit retention policy truth.
- Expired learning audit events are cleaned with parameterized SQL.
- Operators can inspect and trigger retention cleanup.
- Active-account learning content is not auto-deleted by this slice.
