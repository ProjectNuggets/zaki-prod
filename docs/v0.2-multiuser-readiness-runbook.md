# ZAKI v0.2 Multiuser Readiness Runbook

This runbook is the execution sequence for deciding if `rate-limits` is safe to merge into `dev` and deploy against the online Nullalis cluster.

## 1) Preflight (clean + deterministic)

1. Ensure branch is `rate-limits` and working tree is clean.
2. Do not commit generated test artifacts:
   - `playwright-report/`
   - `test-results/`
3. Run local validation first:
   - `npm --prefix backend run lint`
   - `npm --prefix backend run test`
   - `npm run typecheck`
   - `npm test -- --runInBand`
   - `npm run test:e2e`

## 2) Backend env for cluster validation

Set these in the deployed ZAKI backend environment:

```bash
ZAKI_AGENT_BACKEND_ENABLED=true
NULLCLAW_BASE_URL=https://<nullalis-host>
NULLCLAW_INTERNAL_TOKEN=<shared-internal-token>
ZAKI_STREAM_UPSTREAM_TIMEOUT_MS=300000
ZAKI_AGENT_RATE_LIMIT_PER_MINUTE=60
ZAKI_APP_CHAT_DAILY_PROMPT_LIMIT=5
ZAKI_APP_CHAT_DAILY_PROMPT_BUCKET=app_chat
ZAKI_BOT_DAILY_PROMPT_LIMIT=5
ZAKI_BOT_DAILY_PROMPT_BUCKET=zaki_bot
```

## 3) Nullalis cluster alignment checks (from nullalis repo)

Source of truth:
- `nullalis/deploy/k8s/zaki-bot/README.md`
- `nullalis/deploy/k8s/zaki-bot/05-deployment.yaml`
- `nullalis/deploy/k8s/zaki-bot/07-ingress.yaml`
- `nullalis/deploy/k8s/zaki-bot/ZAKI_BACKEND_HANDOFF.md`

Required checks before running multiuser:

1. Ingress has SSE-safe settings:
   - `proxy-read-timeout=3600`
   - `proxy-send-timeout=3600`
   - `proxy-buffering=off`
2. Ingress sticky routing by user id:
   - `nginx.ingress.kubernetes.io/upstream-hash-by: "$http_x_zaki_user_id$arg_user_id"`
3. Nullalis deployment has drain lifecycle hooks enabled.
4. HPA baseline is active (`minReplicas=3`) for staging/prod-like load tests.
5. Internal token and user-id forwarding contract is respected.

## 4) Prepare multiuser tokens for scripted tests

Option A (inline env var):

```bash
export ZAKI_BASE_URL=https://<zaki-backend-host>
export ZAKI_MULTIUSER_TOKENS="token_user_1,token_user_2,token_user_3"
```

Option B (recommended file):

```bash
cat > /tmp/zaki-multiuser-tokens.txt <<'EOF'
token_user_1
token_user_2
token_user_3
EOF

export ZAKI_BASE_URL=https://<zaki-backend-host>
export ZAKI_MULTIUSER_TOKENS_FILE=/tmp/zaki-multiuser-tokens.txt
```

Optional stress tuning:

```bash
export ZAKI_MULTIUSER_ROUNDS=5
export ZAKI_MULTIUSER_P95_TARGET_MS=3000
export ZAKI_MULTIUSER_ERROR_RATE_TARGET=0.005
```

## 5) Execute multiuser suite

Run each step explicitly:

```bash
npm run smoke:agent-multiuser
npm run smoke:agent-isolation
npm run stress:agent-multiuser
```

Or run as one gate:

```bash
npm run smoke:agent-suite
```

## 6) Acceptance gates

Required to pass:

1. Smoke: each user provisions + streams successfully.
2. Isolation: `leaks = 0`.
3. Stress: `p95 <= target` and `errorRate <= target`.
4. No cross-user throttling from same NAT/IP.
5. ZAKI BOT 429 responses include `surface: "zaki_bot"` and reset metadata.
6. Normal app chat remains unlimited for paid/access users.

## 7) Initial safe deployment envelope

Until live metrics confirm higher capacity, use:

1. Start at ~20 concurrent active ZAKI BOT turns per backend pod.
2. Increase to 40–60 only after stress gates pass in cluster.
3. Scale Nullalis by HPA and monitor:
   - lock conflicts
   - stream error ratio
   - p95 latency
   - 503 overload/drain responses

## 8) Merge decision rule

Merge `rate-limits` to `dev` only if:

1. All local automated tests pass.
2. Multiuser suite passes against online cluster.
3. Working tree has no generated artifacts.
4. Nullalis ingress/drain/sticky routing checks are verified.
